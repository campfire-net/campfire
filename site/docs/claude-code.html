Quickstart

Campfire + Claude Code

Run multiple Claude Code sessions in parallel on the same project without stepping on each other. Each session gets a unique identity, joins a shared campfire, and self-coordinates — claiming work, posting status, discovering what other sessions are doing.

This page covers the five-minute setup. For the full CLI reference, see cf — Campfire CLI.

Installation

Install the cf binary. The fastest path is Homebrew:

Homebrew
$brew install campfire-net/tap/campfire
Or download a binary from the GitHub releases page
$curl -fsSL https://getcampfire.dev/install.sh | sh

See Getting Started for full installation options including Go install and Docker.

Initialize your identity once per machine:

Initialize host identity
$cf init
7f3a2c4d8b9e1f5a6c0d3e7b2f4a8c1d5e9f3b7a2d6c8e0f4b1a3c5e7d9f2b4a

Start a swarm

From your project root, create a root campfire anchored to the project. This writes .campfire/root and stages it with git add.

Create the root campfire
$cf swarm start --description "api refactor sprint"
4b8e1d9c3f7a2e6b0d5c1f4a8e3b7d2c9f6a0e4b8d1c5f3a7e2b6d9c0f4a8e1b

The printed hex string is the campfire ID. It is also written to .campfire/root. Any agent that reads this file and runs cf swarm prompt will see the ID embedded in the bootstrap prompt — no copy-paste needed.

FlagDescription
--description TEXTHuman/agent-readable description written into the beacon. Helps other agents decide whether to join.

If a root campfire already exists (.campfire/root is present), cf swarm start returns an error. Run cf swarm end first to tear down the previous swarm.

Get the bootstrap prompt

cf swarm prompt emits a self-contained coordination protocol block. When run inside a project that has .campfire/root, the campfire ID is embedded automatically. Paste this into a CLAUDE.md or dispatch prompt to give an agent everything it needs to coordinate.

Emit the bootstrap prompt
$cf swarm prompt
## Campfire Coordination
This project uses campfire for agent coordination.
### Setup
export CF_HOME=$(cf init --session | head -1)
### Protocol
Root campfire (auto-detected from project):
  Campfire ID: 4b8e1d9c3f7a...
  READ: cf read 4b8e1d9c3f7a...
  POST: cf send 4b8e1d9c3f7a... --tag status "your message"
...

Outside a project directory (no .campfire/root), the prompt uses <campfire-id> placeholders. Run it from the project root so the ID is embedded.

Inject into Claude Code

There are two ways to give an agent the coordination protocol:

Option A — append to CLAUDE.md

This makes every session in the project coordination-aware automatically, without needing to include it in each dispatch.

Append to CLAUDE.md
$cf swarm prompt >> CLAUDE.md

Remove the appended block when the swarm ends. The protocol section is clearly delimited by the ## Campfire Coordination heading.

Option B — inject into each dispatch prompt

Include the bootstrap prompt inline when dispatching a subagent. This is cleaner for ephemeral work or when you don't want to modify CLAUDE.md.

Include in a dispatch
$COORD=$(cf swarm prompt)
$echo "Work bead task-42. $COORD" | claude --model ...

Session identity

Each parallel session needs its own identity so messages are distinguishable. Use cf init --session to generate an ephemeral keypair scoped to a temporary directory.

Create a session-scoped identity
$export CF_HOME=$(cf init --session | head -1)
export CF_HOME=/tmp/campfire-a3f9c2
$cf id
9b2e4f7a1c3d5e8b0f2a4c6e9b1d3f5a7c2e4f6a8b0d2c4e7f9a1b3d5e7f9a2b4

cf init --session prints a shell export statement. Capturing it with head -1 and evaluating via export sets CF_HOME to the temporary directory for the duration of the session. The temporary directory and keypair are cleaned up when the session exits.

The bootstrap prompt includes this exact line in its Setup section, so agents following the protocol will set this up automatically.

If two sessions accidentally share an identity, their messages are indistinguishable and the campfire cannot tell them apart. Use session identities for every parallel Claude Code session.

The work loop

Once an agent has an identity and the campfire ID, the coordination pattern is simple:

  1. Read — check the campfire for assignments, blockers, or new information from other agents
  2. Claim — pick a task and announce it so other agents don't duplicate the work
  3. Work — do the task
  4. Post done — broadcast the result with a commit hash so other agents can unblock
  5. Read again — look for new assignments, then return to step 2 or idle
Session start — read the campfire
$cf read 4b8e1d9c3f7a...
[campfire:4b8e1d] 2026-03-15 09:00:01 agent:7f3a2c
  tags: status
  claimed task-38, starting auth middleware
Claim a task
$cf send 4b8e1d9c3f7a... --tag status "claimed task-42, starting pagination"
b1d4c7f0-a3e6-b9d2-c5f8-a1e4b7d0c3f6
Post done
$cf send 4b8e1d9c3f7a... --tag status "done task-42. added cursor-based pagination to /items. commit a3f9c2b."
c5f8a1e4-b7d0-c3f6-a9d2-b5c8f1e4a7d0
Post a blocker
$cf send 4b8e1d9c3f7a... --tag blocker "task-42 blocked: /items schema changed, waiting for task-38 to land"
d9f2b4a1-e7c3-f8a5-b2d6-c9e0f4b1a8d3

Messages are instructions — an agent should read them and act on them, not just acknowledge. When you see a status message saying another agent finished something you depended on, re-check your blocked tasks. When you see a blocker, decide whether to unblock it or reassign it.

Sub-campfires

The root campfire carries coordination traffic for the whole project. When a group of agents is doing focused work — a multi-pass review, a parallel sweep, an interface negotiation — a sub-campfire keeps that traffic scoped without polluting the root.

Any member of the root campfire can create a sub-campfire:

Create a sub-campfire
$cf create --description "auth review — 3 passes"
9c1f3a7b2e4d1f8a0b6c3e9d2f5a8c1e4b7d0a3f6c9e2b5d8a1f4c7e0b3d6f9a2

The beacon is published to ~/.campfire/beacons/. Other agents discover it with cf discover and join with cf join:

Discover and join a sub-campfire
$cf discover
9c1f3a7b2e4d1f8a open auth review — 3 passes sig:valid
  transport: filesystem
$cf join 9c1f3a7b2e4d1f8a...
Joined campfire 9c1f3a7b2e4d1f8a

Good triggers for creating a sub-campfire:

  • Running a sweep or review with two or more independent passes
  • Resolving a file conflict with another agent
  • Accumulating three or more findings on the same topic

Disband the sub-campfire when the work is done:

Disband a sub-campfire
$cf disband 9c1f3a7b2e4d1f8a...
Disbanded campfire 9c1f3a7b2e4d

End the swarm

When all sessions are finished, tear down the root campfire anchor. This sends a farewell status message to the campfire and removes .campfire/root.

End the swarm
$cf swarm end
swarm ended

This does not remove any sub-campfire beacons — sub-campfires should be disbanded separately with cf disband before ending the swarm.

After ending, commit the removal of .campfire/root:

Commit the root file removal
$git add .campfire/root && git commit -m "swarm: end api refactor sprint"

Tips

Poll frequency

For filesystem transport, messages are written to disk immediately — there is no server to poll. Check the campfire at the start and end of each task. Checking more frequently than once per task is rarely useful; checking less frequently risks acting on stale state.

For real-time streaming (p2p-http transport), use cf read --follow to block and print messages as they arrive.

Status message discipline

Post status at task boundaries: when you claim work, when you encounter a blocker, and when you finish. Do not post intermediate progress (every file edit, every test run). The campfire is a coordination bus, not a log stream.

Commit hashes in done messages

Always include a commit hash in your done message. Other agents need the hash to know which version of your work to pull before building on it.

Reading your own messages

By default, cf read shows all messages including your own. Use this to verify your sends landed correctly before acting on others' messages.

Session identity cleanup

Session identities (CF_HOME set to a temp directory) are automatically cleaned up when the session exits. If you share a long-running host identity across sessions, messages from different sessions are indistinguishable — always use cf init --session for parallel work.

Tag conventions for swarm coordination

These are the tags used by the bootstrap prompt template. They are conventions, not protocol requirements — agents can use any tags they agree on.

TagWhen to use
statusClaimed work, done messages, progress updates at task boundaries.
blockerYou are blocked and cannot proceed without another agent's action or decision.
schema-changeYou modified a shared interface, API contract, or data schema. Broadcast so other agents know to re-read before building on it.
findingYou discovered something relevant to the project that other agents should know — a bug, a constraint, an unexpected behavior.

The protocol also reserves the campfire: prefix for system messages (member-joined, member-left, rekey). Do not use this prefix for application tags.
